# 데이터 신뢰도 정책

이 문서는 제품 정보의 출처, 검수 상태, AI 추정값을 화면에 표시하는 기준을 정의한다. 목표는 사용자가 점수뿐 아니라 정보의 신뢰 수준을 함께 판단하게 하는 것이다.

## 원칙

- 브랜드, 제조사, 광고주가 점수나 추천 순위에 영향을 주면 안 된다.
- 추천 영역은 후원 상품이나 광고 상품을 포함하지 않는다.
- 사용자 제보와 AI 추정값은 검수 완료 데이터처럼 표시하지 않는다.
- 출처가 없는 건강 주장은 표시하지 않는다.
- 개인정보와 스캔 이력은 상업적 목적으로 판매하거나 공유하지 않는다는 원칙을 앱 정보 화면에 명시한다.

## 출처 등급

| 등급 | 출처                    | 화면 표시   | 점수 사용      |
| ---- | ----------------------- | ----------- | -------------- |
| T1   | 자체 검수 DB `verified` | 검수 완료   | 확정 평가      |
| T2   | 식약처·공공데이터       | 공공데이터  | 확정 평가 가능 |
| T3   | Open Food Facts         | 공개 DB     | 참고 평가      |
| T4   | AI 구조화               | AI 추정     | 임시 평가      |
| T5   | 사용자 직접 등록        | 사용자 제보 | 검수 전 평가   |

T4, T5는 제품 상세와 검색 결과에서 `검수 전` 또는 `임시` 표시를 반드시 붙인다.

## 바코드 조회 후보 선택

바코드 제품 확인은 자체 DB, 공공데이터, 공개 DB, AI 보강 4단계를 모두 실행한 뒤 종료한다. 자체 DB 또는 공공데이터에서 후보 1개가 확인되어도 즉시 반환하지 않는다.

Gemini는 `selectBestProductCandidate` action에서 후보 중 하나의 index만 선택한다. 제품명, 영양성분, 원재료를 새로 생성하거나 여러 후보를 병합하지 않는다. Gemini 호출이 실패하면 검수 완료, 공공데이터, 공개 DB, 임시 데이터 순으로 deterministic fallback을 적용하고 화면에는 AI 단계 실패를 표시한다.

`translateProductToKorean` action은 외국어 공개 DB 또는 AI OCR 텍스트의 표시 언어만 한국어로 바꾼다. 이 action은 영양성분, 바코드, 출처, 신뢰도, 회수 정보, 첨가물 코드를 변경하지 않으며, 번역 실패 시 원문을 그대로 표시한다.

`explainAdditiveTerms` action은 사전에 없는 첨가물 코드나 전문용어를 사용자가 이해하기 쉬운 한국어 이름, 사용 목적, 한 문장 설명으로 바꾼다. 이 action은 안전성 판정, 위험도, 점수, 금지 여부를 생성하지 않으며 결과는 제품 상세 화면에서 `AI 해석`으로만 표시한다.

## 영양성분 출처

공식 가공식품 영양 DB를 선적재한 `nutrition_products`는 T2 공공데이터로 취급한다. 바코드 자체가 아니라 식품안전나라 C005/I2570에서 얻은 품목제조보고번호, 제품명, 제조사명으로 매칭한다.

매칭 우선순위:

1. 품목제조보고번호 exact match
2. 정규화 제품명 exact match
3. 제조사/브랜드 접두어를 제거한 정규화 제품명 exact match
4. 정규화 제품명 partial match
5. 제조사/브랜드 접두어를 제거한 정규화 제품명 partial match
6. 공공데이터포털 영양성분 API fallback

품목제조보고번호 없이 제품명만으로 매칭된 경우 동명이품 가능성이 있으므로 confidence와 후보 선택 결과를 함께 고려한다. C005/I2570에서 바코드가 확인되지 않고 Open Food Facts에서 제품명만 확인된 제품도 `enrichProduct`가 `nutrition_products` 제품명·제조사 매칭으로 영양성분을 보강할 수 있다. 이 경우 보강값은 공식 영양 DB 행에서만 가져오며, 기존 후보의 열량·당류·단백질과 원재료 맛 힌트를 후보 랭킹에 반영한다. 복수 후보가 있거나 품목명이 넓으면 검수 대상으로 남긴다.

## 제품 상태 표시

| 상태              | 의미             | 사용자 문구     |
| ----------------- | ---------------- | --------------- |
| `verified`        | 관리자 검수 완료 | 검수 완료       |
| `public_matched`  | 공공 API 매칭    | 공공데이터 기반 |
| `open_db_matched` | 공개 DB 매칭     | 공개 DB 기반    |
| `provisional`     | 임시 저장        | 임시 분석       |
| `needs_review`    | 검수 필요        | 검수 필요       |

`verified`가 아닌 제품은 점수 옆에 신뢰도 배지를 함께 표시한다.

## 신뢰도와 점수 표시

| 조건                               | 표시 방식                      |
| ---------------------------------- | ------------------------------ |
| T1 또는 T2이고 필수 영양 정보 충분 | 숫자 점수 표시                 |
| T3이고 필수 영양 정보 충분         | `참고 점수`로 표시             |
| T4 또는 T5이고 confidence 0.7 이상 | `임시 점수`로 표시             |
| T4 또는 T5이고 confidence 0.7 미만 | 숫자 점수 숨김, 검수 필요 표시 |
| 필수 영양 정보 부족                | `평가 불가` 표시               |

confidence는 데이터 추출 신뢰도다. 건강성 점수와 혼동되면 안 된다.

## 검수 큐 적재 기준

다음 조건 중 하나라도 만족하면 검수 큐에 적재한다.

- AI OCR confidence가 0.7 미만
- 사용자 직접 등록 제품
- 미분류 첨가물이 있음
- 회수·판매중지 이력 매칭이 불확실함
- 동일 바코드에 서로 다른 제품명이 존재함
- 필수 영양성분이 일부만 추정됨

## 대체상품 추천 신뢰도

대체상품 추천은 다음 순서로 우선한다.

1. 공공 영양 DB에서 점수 산출 가능한 후보군을 수집한다. 이때 분류·품목명 조회는 후보 수집용이며 최종 추천 기준으로 쓰지 않는다.
2. Gemini `recommendAlternativeProducts`는 후보를 새로 만들지 않고, 공공 DB/API 후보의 실제 대체 가능성과 영양 개선 가능성을 기준으로 추천 순서를 판정한다.
3. Gemini 판정은 좁은 제품 형식, 섭취 경험, 구매 의도, 사용 맥락, 조리·보관 방식, 제품명을 기준으로 하며 키워드 일치나 넓은 식품군 일치만으로 통과시키지 않는다.
4. Gemini가 추천 목록에 포함하고 `fitScore`가 70점 이상인 후보만 추천 평가에 사용한다.
5. 검수 완료, 공공데이터, Open Food Facts처럼 점수 산출 가능한 출처만 사용한다.
6. 후보 점수는 B등급 기준인 70점 이상이어야 한다.
7. 현재 제품의 점수를 산출할 수 있으면 최소 5점 이상 높아야 한다.
8. 당류, 나트륨, 포화지방, 열량, 첨가물 수 중 최소 1개 부담이 낮아야 한다.
9. 회수·판매중지 이력이 확인된 후보는 제외한다.
10. 동일 바코드, 동일 품목제조보고번호, 정규화 상품명 일치, 또는 같은 브랜드·카테고리 맥락에서 짧은 핵심 상품명이 포함되는 동일 제품 라인은 기본 대체상품에서 제외한다.

공공 영양 DB 후보가 없거나 Gemini가 후보 목록에서 실질 대체상품을 고르지 못하면, Gemini가 현재 제품의 영양 부담을 해석해 네이버쇼핑 검색어를 생성한다. 그 뒤 네이버쇼핑 상위 결과를 Gemini가 다시 검토해 검색 의도와 영양 대체 방향에 맞는 상품만 통과시킨다. Gemini는 검색어, 통과 상품 id, 영양 방향만 만들며 실제 상품명, 가격, 이미지, 링크는 네이버쇼핑 API 결과만 사용한다.

Gemini는 다음 필드만 반환한다.

- `baseSubstituteGroup`
- `productId`
- `isSubstitute`
- `fitScore`
- `substituteGroup`
- `reason`
- `warnings`

Gemini는 제품명, 영양성분, 가격, 출처를 생성하거나 수정하지 않는다. Gemini 호출이 실패하면 추천을 표시하지 않는다. 네이버쇼핑 경로에서 표시되는 상품은 기본적으로 가격·판매처 보조 정보와 Gemini의 영양 방향만 표시한다. Open Food Facts에서 상품명 기반 영양성분 보강이 성공하고 점수 기준을 통과한 경우에만 참고 점수와 등급을 표시한다.

추천 사유는 반드시 표시한다.

예:

- `당류가 더 낮습니다`
- `나트륨이 더 낮습니다`
- `고위험 첨가물이 없습니다`
- `국내 구매 가능`

추천 후보가 부족하면 `추천할 만큼 신뢰할 수 있는 대체상품이 없습니다`를 표시한다.

네이버쇼핑 검색 결과는 구매 가능성, 가격 텍스트, 판매처, 링크 보강에 사용한다. 공공 영양 DB 기반 점수형 후보가 없을 때는 Gemini가 만든 영양 대체 검색어로 네이버쇼핑을 조회하고 Gemini 재검증을 통과한 결과만 표시한다. Open Food Facts는 네이버쇼핑 통과 상품의 영양성분 보조 검증에만 사용하며, 상품 후보를 직접 채우는 fallback으로 사용하지 않는다. 가격은 후보 간 정렬 기준으로 사용하지 않는다.

## 함께 먹기 좋은 조합 추천 신뢰도

함께 먹기 좋은 조합은 제품의 영양성분을 기준으로 Gemini `recommendFoodPairings` action이 생성한다.

적용 기준:

1. 영양성분 값이 하나도 없으면 Gemini를 호출하지 않고 근거 부족 문구를 표시한다.
2. Gemini는 제품 데이터를 수정하거나 새 제품 점수, 가격, 구매 링크를 생성하지 않는다.
3. 추천 대상은 브랜드 상품이 아니라 일반 식품 또는 간단한 식품 조합이다.
4. 추천 근거는 단백질, 식이섬유, 수분, 당류, 나트륨, 포화지방, 열량 균형처럼 일반 영양 기준으로 제한한다.
5. 질병 치료, 체중 감량, 의학적 진단, 개인 맞춤 치료 효과 주장은 표시하지 않는다.
6. 알레르기 또는 라벨 확인이 필요한 경우 주의 문구를 함께 표시한다.

Gemini는 다음 필드만 반환한다.

- `overallStrategy`
- `foods`
- `reason`
- `nutritionFocus`
- `caution`
- `fitScore`
- `warnings`

조합 추천은 대체상품 추천과 별도 영역으로 표시하며 후원 상품이나 광고 상품을 포함하지 않는다.

| 화면      | 필수 표시                                                                                  |
| --------- | ------------------------------------------------------------------------------------------ |
| 홈        | 최근 스캔 제품의 점수 색상, 검수 상태                                                      |
| 검색 결과 | 점수, 위험 배지, 출처 배지                                                                 |

| 설정      | 개인 기준 진입점, 독립 점수 원칙, 비후원 추천 원칙, 기록 비판매 원칙, 로컬 데이터 인벤토리 |

## 신뢰 센터

`/settings`는 개인 기준 진입점과 다음 원칙을 사용자에게 직접 표시한다.

- 브랜드, 제조사, 광고주는 점수와 판정에 영향을 줄 수 없다.
- 대체상품 추천은 점수와 위험 요인 개선 기준으로만 정렬하며 후원 상품을 포함하지 않는다.
- AI OCR과 사용자 제보 데이터는 검수 완료 데이터처럼 표시하지 않는다.
- 개인정보와 스캔 이력은 상업적 목적으로 판매하거나 공유하지 않는다.
- 제품 정보는 의학적 진단을 대체하지 않는다.

설정 화면은 로컬 저장 항목의 개수와 목적도 함께 표시한다. 최근 스캔, 저장 제품, 임시 제품, 개인 기준, 임시 초안, 최근 검색어는 모두 브라우저 로컬 저장소에 보관되며 `로컬 데이터 모두 삭제`로 삭제된다.

`로컬 데이터 내보내기`는 위 항목을 `food-scan-local-data-YYYYMMDD.json` 파일로 생성한다. `로컬 데이터 가져오기`는 앱 식별자와 버전을 검증한 뒤 기존 로컬 데이터를 백업 파일 내용으로 대체한다. 세부 스키마와 최대 보존 개수는 [로컬 데이터 이동 정책](./local-data-portability-policy.md)을 따른다.
